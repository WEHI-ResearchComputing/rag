Milton dashboards

This guide introduces the Milton dashboards and describes the different dashboards available.

The dashboards can be found at dashboards.hpc.wehi.edu.au, and were built using Grafana, Prometheus and Go. They are developed and maintained by the RCP team with the support of the ITS Systems team. Dashboards will be added as the need arises.

Please note that you need to be logged in to WEHI's network or VPN (Global Protect) for the link to activate.

A new dashboard was recently added and can be found on the FileSystem tab: VAST Scratch Area Utilisation.

Currently, there are four dashboards.

Available dashboards

The four available dashboards are:

  • FileSystem
  • RStudio Servers
  • SLURM
  • User Queue Details.

The image gallery below shows a screenshot of each dashboard with a brief explanation of its features. Click the arrow on the mid-right of each image to scroll through the gallery and see a screenshot (with accompanying information) of all 4 dashboards.

(1) The FileSystem dashboard

Shows the utilisation percentage of each partition on the different filesystems. There are two vertical tabs, VAST and Stornext, which can be expanded or collapsed.

(2) RStudio Servers dashboard

This dashboard shows different utilisation of the two RStudio servers with respect to CPU, Memory and /tmp. You can also see a per user utilisation.

(3) The SLURM dashboard

This dashboard has two tabs: 

  1. SLURM Cluster
  2. SLURM Jobs.


The SLURM Cluster tab shows an overall view of the SLURM CPUs, Memory and GPUs utilisation.

The SLURM Jobs tab gives a snapshot of the state of the SLURM queue currently.

Availability of resources determines whether a job starts running or is held in a pending state. On Milton SLURM, multi-factor priority scheduling with backfill is used to assign job priority which affects when the job will run.

The regular QOS currently limits a single user to 256 CPU cores and 1.3TB memory, even if more resources are available. Jobs pending due to QOS are shown on the tab under QOS Pending.

Other parameters (jobs pending due to other factors are shown under Pending) are :
​​​​​​​

  • Fair share: the more resources your jobs are already using, the lower the priority of your next jobs.
  • Age: the length of time a job has been waiting in the queue, as it increases the job priority increases.
  • Resources requested: the more resources (CPUs, GPUs and memory) requested, the higher the priority.

(4) User Queue Details

This dashboard presents a snapshot of the queue per user.

This is why you have to choose a user to see that user's queue information, just as if you run the squeue -u <username> command.

It has two tabs:

  1. Running Jobs
  2. Other Jobs.


​​​​​​​The Running Jobs tab shows the number of jobs running currently and the total amount of resources allocated to these running jobs.

The Others tab shows the number of jobs not running (i.e. pending, suspended,..) currently and reason why they are not running, as show in the example above.
